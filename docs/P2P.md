# P2P-туннель (замена Radmin/Hamachi)

Модуль [`backend/p2p/`](../backend/p2p) даёт прямое подключение между друзьями
без ручного проброса порта, через NAT любого типа (кроме двойного Symmetric NAT —
для него нужен relay-сервер, см. ниже).

## Поток установления сессии

```
Хозяин (А)                                    Друг (Б)
   |                                              |
   | go_online() -> ensure_identity()             | go_online() -> ensure_identity()
   | (генерируется Curve25519-ключ, если его нет) |
   |                                              |
   |        <— подписка на свой topic в MQTT —>   |
   |                                              |
   |                          connect_to_friend(А)|
   |                          gather_candidates()  | (host IP + STUN srflx)
   |              <— offer (pub key, candidates) — |
   |  (проверка: А есть в списке друзей Б? Да)     |
   |  gather_candidates(), answer ——————————————>  |
   |                                              |
   |  UDP hole punching на все кандидаты ←———————→ | (simultaneous open)
   |                                              |
   |  SessionCrypto = DH(my_priv, peer_pub)        | то же самое
   |  ReliableUdpChannel (ack/retransmit/keepalive)|
   |                                              |
   |  127.0.0.1:25565 (реальный MC-сервер)         | 127.0.0.1:<local_port> (видит Minecraft)
```

## Почему это безопасно

- **Авторизация** — offer от человека, которого нет в `config.json["friends"]`,
  тихо игнорируется. Никакой произвольный пользователь публичного MQTT-брокера
  не может инициировать сессию.
- **TOFU-пиннинг ключа** — публичный ключ друга запоминается при первом обмене
  (как `known_hosts` в SSH). Если в следующий раз придёт offer с другим ключом
  под тем же именем — он отклоняется (защита от подмены идентичности).
- **Шифрование** — каждый пакет туннеля шифруется `nacl.SecretBox` с ключом,
  выведенным через Diffie-Hellman между постоянными ключами двух сторон.
  Перехватчик трафика (даже сам STUN/MQTT-сервер) видит только шифротекст и
  не может ни прочитать, ни подменить содержимое — IP-спуфинг бесполезен без
  приватного ключа.
- **Сигнальный канал** (MQTT) публичный, но по нему не передаётся ничего, кроме
  кандидатов и публичных ключей — сам игровой трафик никогда не проходит через MQTT.

## Производительность

- Туннель — это надёжный канал поверх "сырого" UDP (см. `reliable_udp.py`):
  собственный ACK/retransmit без оверхеда полного TCP/IP стека.
- Keep-alive (ping/pong) каждые 5 секунд держит NAT-маппинг живым — большинство
  роутеров закрывают UDP "сессию" после 20–60 секунд простоя.
- При обрыве пути (`on_dead`) автоматически запускается reconnect с backoff
  (1/2/4/8/8 секунд) — короткая потеря пакетов не выкидывает игрока с сервера.

## Когда P2P не работает

Если оба игрока находятся за **Symmetric NAT** (типично для некоторых
мобильных операторов и корпоративных сетей), hole punching принципиально не
срабатывает — внешний порт меняется на каждое новое направление трафика.
В этом случае нужен TCP relay-фолбэк (`turn_relay.py`):

```powershell
# на любом сервере с публичным IP (VPS за $3-5/мес)
python -m backend.p2p.turn_relay 27460
```

Relay видит только шифротекст (end-to-end шифрование не снимается), поэтому
один relay можно безопасно использовать для любого числа пар друзей.
Адрес relay-сервера передаётся в `P2PManager(..., relay_server=("host", 27460))`.

## Известные ограничения текущей реализации

- Один P2P-канал хоста обслуживает одного гостя за раз (нет мультиплексирования
  нескольких одновременных подключений на одном порту) — для пары друзей это
  не проблема, для сервера на 5+ человек нужна доработка `manager.py`.
- NAT-PMP (альтернатива UPnP на части роутеров) не реализован — используется
  только UPnP (`backend/upnp.py`) как запасной путь, P2P-туннель его не требует.
- Классификация NAT (`stun.detect_nat_type`) грубая (RFC 3489-style) и не
  заменяет полноценный ICE/TURN стек уровня WebRTC — для домашних сетей
  (Full Cone / Restricted Cone / Port-Restricted Cone) этого достаточно.
